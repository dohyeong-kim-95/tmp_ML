# 10 — Q9: 표현 파이프라인(X0→…→Xk(latent)→…→Xn(shmoo)→Y) 타당성 평가

> 제안: **X0(60bit) → X1(룰베이스 도메인지식으로 살짝 차원축소) → … → Xk(잠재공간) → … → Xn(shmoo) → Y(metric)**
> 으로 표현을 단계화해 도메인지식을 유연하게 주입.

## 결론 먼저
**방향성은 타당하다.** "표현(도메인지식)과 옵티마이저를 분리하고, 지식을 *교체형 변환 레이어*로 점증 주입"한다는 발상은
실제 기법(인코딩 기반 제약, 잠재공간 최적화/LSBO)과 정합한다. 단 **전 구간을 한 번에 모델링하려 하지 말고**,
**단계별로 필요한 부분만** 쓰는 게 좋다. 일부 단계(학습된 잠재공간)는 **데이터가 쌓인 뒤** 도입 → 현 단계 보류.

## 단계별 타당성

| 구간 | 성격 | 평가 | 도입 시점 |
|------|------|------|-----------|
| **X0 → X1** (룰베이스 차원축소) | 배타/그룹/조건부를 규칙으로 합성·축소 | ✅ **강력 추천**. 유효차원↓·불가능조합 제거. 이미 §5.1 Phase0/research-04의 "인코딩 슬롯"과 동일 | 지금부터(규칙 확보되는 대로) |
| **X1 → … → Xk** (학습된 잠재공간) | VAE/오토인코더 등으로 연속 잠재표현 학습 후 그 위에서 최적화(LSBO) | ⚠️ **조건부 추천**. 이산·고차 조합을 연속공간으로 바꿔 표준 대리모델/획득 사용 가능. 단 **유효성(decode 시 유효·비중복) 문제**와 **데이터 요구**가 큼 | 데이터 충분(수만) + 필요성 확인 후 |
| **Xk → … → Xn** (shmoo array 예측) | 중간 산출물(shmoo)을 명시적으로 예측 후 metric 도출 | ⚠️ **선택적**. 풍부한 신호·멀티태스크 이점 가능하나 모델 부담↑. metric만 필요하면 생략 가능 | 후순위(여력/이점 확인 시) |
| **Xn → Y** (rule-base metric) | shmoo→metric은 **알려진 규칙** | ✅ 규칙이므로 학습 불필요. **알면 명시적으로 끼움** | 규칙 확보 시 |

## 왜 타당한가 (근거)
- **인코딩/제약 레이어(X0→X1)**: 배타·그룹을 범주형으로 재파라미터화하면 차원·불가능조합을 동시에 줄임(research/04).
  이는 "도메인지식 = 입력 변환"으로 보는 본 제안의 핵심과 일치.
- **잠재공간 최적화(LSBO)**: 큰 이산/조합 공간을 **연속 잠재공간 Z로 매핑 → 표준 대리모델·획득함수 사용 → 디코더로 복원**.
  분자·서열 설계 등에서 표준화된 접근. 본 제안의 "Xk(잠재공간)"과 정확히 대응.
- **합성 가능 구조의 운영 이점**: 각 변환을 **교체형 모듈**로 두면, 새 도메인지식이 생길 때마다 **레이어를 끼우거나 교체**만 하면 됨 → "프로젝트 기간 내내 지식 점증 주입"이라는 요구에 부합.

## 주의/리스크
- **LSBO 유효성 문제**: 디코딩 결과가 자주 **무효/중복**일 수 있음 → 디코더 불확실성 활용, 유효성 제약 필요.
- **잠재공간 품질**: 고차원·구조 불량 잠재공간은 오히려 대리모델을 어렵게 함 → 저차원·정규화·반지도 학습 권장.
- **과설계 경계**: 전 구간(shmoo까지)을 모델링하면 복잡도·실패점↑. **"필요한 최소 표현"** 원칙 유지.
- **데이터 의존**: 학습된 잠재공간은 수만 표본·검증이 필요 → 현 단계는 **인터페이스(슬롯)만 설계**, 구현은 데이터 후.

## 권고 설계 (현 단계에서 할 일)
- 파이프라인을 **합성 가능한 transform 인터페이스**로 추상화:
  `raw(X0) → [Encode(규칙)] → [LatentMap(학습, 선택)] → repr` 그리고
  `repr → Surrogate/Optimizer`, 별도로 `shmoo → metric(규칙)`.
- 지금은 **Encode(규칙) 단계와 인터페이스만** 구현(규칙은 비어 있어도 항등변환).
  **LatentMap·shmoo예측은 슬롯으로 비워둠** → 데이터·필요 확인 후 채움.
- 이렇게 하면 옵티마이저(MAIN/SUB)는 `repr` 위에서 동작 → **표현이 바뀌어도 옵티마이저 불변**.

## 이 파이프라인이 제약 처리(#4)를 일부 해결한다
- group/mutex/conditional은 **X0→X 인코딩 단계에서 흡수**된다(그룹→범주형, 배타→범주형, 조건부→계층).
- 따라서 **DoE·옵티마이저는 raw X0가 아니라 인코딩된 X에서 동작**해야 한다(research/03 §제약 하 DoE).
  - PB 등 직교설계를 raw 비트에 쓰면 독립요인 가정이 깨지는 문제가, **X 단계로 옮기고 제약-aware D-optimal을 쓰면 해소**.
- 즉 표현 파이프라인은 "도메인지식 주입 경로"이자 동시에 "**실험설계가 합법 요인공간에서 이뤄지게 하는 기반**".

> 한 줄 평: **"도메인지식을 입력 변환 레이어로 점증 주입"하는 골격은 타당하고 권장.**
> 단 **학습된 잠재공간/shmoo 예측은 데이터 확보 후 단계적으로** 도입(현재 보류, 인터페이스만 마련).
> **DoE/옵티마이저는 인코딩된 X 단계에서 동작**시켜 제약 문제(#4)를 구조적으로 해결.
