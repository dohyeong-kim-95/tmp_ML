## 0. 문제 한눈에 (recap)
- **N>60 binary X → rule-base metric Y 최대화/목표달성.** 탐색공간 2^60(≈10^18).
- **데이터 0건**에서 시작, 평가는 **비싸고 배치**(MAIN 320/일·SUB 40/일), 15주 누적 **MAIN O(10^4)·SUB O(10^3)**.
- 변수에 **그룹/배타/조건부/큰 상호작용** 구조(규칙은 추후 주입). **노이즈** 있음. **추론 ≤3s(목표 500ms)**.

## 1. 전략 선택 논리 — 제약이 방법을 어떻게 결정하는가 ★
> 이 한 표가 "왜 이 전략인가"의 핵심. 각 제약이 곧장 설계 선택으로 이어진다.

| # | 제약 (01_Problem_def §3·§5) | 함의 | **전략 선택** |
|---|------|------|----------------|
| 1 | 데이터 0건 (cold start) | 초기 데이터를 직접 만들어야 | **실험계획법(DoE) 시드** + 능동적 반복 루프 |
| 2 | 평가 비쌈·배치(40/320 per day) | 표본효율 + 배치 친화 | 저예산=**BO**, 중예산=**SAEA**, 배치는 **다양성 보장 추출** |
| 3 | 데이터 누적 | 반복 갱신 가능 | **순차 모델기반 최적화 루프**(매일 모델 갱신) |
| 4·5 | 예산 ≪ 2^60 | 전수·무작위 불가, **구조 활용 필수** | **구조(상호작용/linkage)를 학습·활용**하는 모델 |
| — | **N>60 binary** | 표준 GP-BO의 컴포트존(연속 20~30D) 밖 | **트리/선형/연관학습 EU** (GP-BO는 보조) |
| 6 | 구조 규칙 추후 주입 | 지금은 구조를 *발견*해야 | 상호작용·linkage를 **데이터로 추정**하는 모델 |
| 7 | 반복 wall-clock | 매일 배치 생성이 빨라야 | **데이터에 선형 확장**되는 모델(트리/선형/EA), GP-HMC·O(n³) 회피 |
| 8 | 추론 ≤3s(목표 500ms), 학습 무제한 | 무거운 탐색은 오프라인으로 | **학습/추론 분리 — 후보 큐**(§5) |
| 9 | 측정 노이즈 + 선택편향 | winner's curse 위험 | **confirmatory 2단계 판정** + 노이즈-aware/robust |
| — | 성공기준 = threshold 통과 | 순수 max 아님 | **satisficing/level-set** 지향 탐색 |
| — | 도메인지식 점증 주입 | 유연한 확장 필요 | **교체형 표현 파이프라인 + 제약 슬롯** |

**한 줄 결론**: 이것은 *"고차원 N>60 binary 조합에 대한, 비싸고 노이즈 있는 배치 black-box 최적화"* 이며,
**예산 체급에 따라** ① **구조를 학습·활용하는 모델기반 탐색**(중예산 MAIN)과 ② **표본효율 우선의 베이지안 최적화**(저예산 SUB)로 갈린다.
공통적으로 **DoE 시드 → 순차 갱신 루프 → 후보 큐 서빙 → confirmatory 판정 → 도메인지식 주입**의 골격을 쓴다.

## 2. 체급별 주력 방법 (현 가설 — Phase 2에서 검증 대상)
| Regime | 누적 예산 | 지배 요인 | **주력 방법** |
|--------|-----------|-----------|----------------|
| 🟢 **MAIN** (320/일) | O(10^4) | 구조 활용 + 알고리즘 효율 | **모델기반 연관학습 진화알고리즘(DSMGA-II / LT-GOMEA) + 지도학습 대리모델(SAEA)** |
| 🔵 **SUB** (40/일) | O(10^3) | 표본 효율 | **베이지안 최적화(BOCS형 희소회귀 / SMAC3 / TPE) + screening DoE** |

> **왜 MAIN은 연관학습 EA인가**: 예산이 O(10^4)면 진화탐색이 본령에 들고, "binary 묶음(building block)" · "큰 상호작용" 구조는
> linkage(연관)를 *학습*하는 EA(DSMGA-II/LT-GOMEA)가 정확히 겨냥한다. 학습된 linkage는 **구조 추정치(가설)** 가 되어 도메인지식화의 다리가 됨.
> **왜 SUB는 BO인가**: 데이터 0건+점당 비용 큼 → 대량 라벨 지도학습/그리드 불가 → **평가 예산을 능동 배분하는 BO**가 본질적으로 맞음.
> **공통**: 두 체급 모두 **공통 인프라**(인코딩/제약 슬롯, 데이터 스키마, 시뮬레이터, 모니터링) 위에서 **옵티마이저만 교체**.

## 3. 단계별 파이프라인

**Phase 0 — 인코딩 & 제약 표현 (도메인지식 슬롯, 공통)**
- binary → 모델 입력 표현으로 매핑. **배타/그룹 → 범주형 재파라미터화**, **조건부 → 계층(DAG)** 공간.
- 구조 규칙은 가능해지는 대로 제약/합성변수/사전지식으로 주입할 **슬롯만 비워둔다**.

**Phase 1 — 초기 데이터 확보 (cold start)**
- ⚠️ **DoE는 raw X0(60bit)가 아니라 인코딩된 feature space X에서 수행.**
  group/mutex/conditional이 있는 raw 비트는 "독립 2수준 요인" 가정을 깨므로, **먼저 X0→X 인코딩 후 합법 요인공간 X에서 설계**.
- 제약이 있으면 PB/분수계획(직교설계)을 그대로 쓰지 말고 **제약-aware 최적설계(D-optimal, coordinate-exchange)** —
  직교설계에 repair를 가하면 균형/별칭구조가 깨짐. **DoE 목표 = 활성 주효과 솎기 + 상호작용 후보 발굴**(완전 추정은 루프로 이연).
- MAIN: 첫 며칠은 (제약-aware) DoE + 광역 무작위로 **수천 표본**을 모아 대리모델 토대 구축. 일부는 **반복측정**(노이즈 추정).
- 세부(candidate set·균형성·screening 범위·예산) → research/03 §4A–4D, feasible pool 생성 → research/04 §4.2.

**Phase 2 — 주력 최적화 루프 (체급별)**
- 🟢 **MAIN — 모델기반 진화 + 대리모델 보조 (SAEA)**
  - **연관학습 EA(DSMGA-II / LT-GOMEA)** 로 변수 묶음(building block) 단위 탐색.
  - **지도학습 대리모델(GBM/RF)** 으로 EA 후보를 **사전선별(triage)** → 상위 320만 실측.
    이때 **순수 top-k 금지 → 다양성(DPP/클러스터/ε-탐색) 동반**(winner's curse·다양성 붕괴 방지).
  - 매일: 후보 생성 → 대리모델 선별 → 320 실측 → 데이터 누적 → 모델·linkage 갱신.
- 🔵 **SUB — 배치 베이지안 최적화**
  - 대리모델: (해석형) **BOCS형 희소 pairwise 베이지안 회귀** / (강건형) **SMAC3(RF)·Optuna(TPE)**.
  - 배치 추출: **배치 Thompson sampling**(q=40에 적합) / Local Penalization / Kriging-Believer.
    (독립 ask×40은 중복/근접 후보 발생 → 금지.)

**Phase 3 — 도메인지식 주입 (발전 단계, 공통)**
- 구조 규칙을 제약/사전분포(prior)/커스텀 feature/커널/EA 연관모델 초기화로 주입.
- effect heredity 등으로 상호작용/연관 탐색을 가지치기. 표현 파이프라인(research/10)에 새 레이어로 끼움.

## 4. 후보 도구 스택 (네트워크/OSS 사용 가능 전제)
| 용도 | 가벼운 baseline | 본격 |
|------|----------------|------|
| MAIN 옵티마이저 | GA(`DEAP`) + 휴리스틱 linkage | **DSMGA-II / LT-GOMEA(GOMEA)** 공개 구현, hBOA |
| MAIN 대리모델 | `scikit-learn` RF | **XGBoost/LightGBM** + SHAP 해석 (불확실성 필요시 CQR/NGBoost 보정) |
| SUB BO | Optuna (TPE, ask/tell 배치) | SMAC3(RF), BoTorch/Ax, BOCS(`BOCSpy`) |
| DoE | pyDOE2 | 커스텀 D-optimal (coordinate-exchange) |

> ⚠️ **반복 시간 가드레일**: 매일 배치(MAIN 320, SUB 40) 생성 wall-clock 점검(목표 배치당 수 분 내).
> GP-HMC(SAASBO)·대형 GP-BO 등 n³/HMC 비용 방법은 N>60·누적 시 부담 → 주력 제외, 비교용으로만.

## 5. 서빙 아키텍처 — 학습/추론 분리 (추론 ≤ 3s, 목표 ≤ 500ms)
> 핵심: **무거운 탐색 = 학습(오프라인, 무제한)**, **다음 60-bit 신호 방출 = 추론(온라인, 하드 한도)** 로 분리.

- **패턴 A — 후보 버퍼/큐 (1순위, 방법 무관)**: 오프라인 옵티마이저가 후보를 미리 생성·정렬해 큐에 적재 → 서빙은 **pop만(~ms)**. 매일 배치 생산과 천연 궁합. 데이터 누적 시 백그라운드 재생성.
- **패턴 B — amortized 추론 모델**: 학습된 정책/대리모델이 상태를 받아 **단일 forward pass로 후보 생성**(보고 1~4ms). 상태반영 즉시생성이 필요할 때.
- **패턴 C — anytime 지역탐색**: GBM 위에서 **500ms 예산** 지역탐색, 시간초 시 best-so-far 반환.

> **권고: 공통으로 패턴 A(후보 큐)** 기본 채택 → 어떤 옵티마이저든 추론 한도 자동 충족. 필요 시 B/C 추가.

## 6. 왜 다른 방법이 아닌가 (기각/보조 처리한 대안)
- **전수/그리드/랜덤 서치**: 2^60 + O(10^3~4) 예산에서 표본효율 너무 낮음 → ❌.
- **순수 지도학습(대량 라벨 선확보)**: 데이터 0건·점당 비용 큼 → 전제 불성립 → ❌(대리모델은 *루프 내*에서만 사용).
- **표준 GP-BO / SAASBO(GP-HMC)**: N>60·데이터 누적 시 O(n³)·HMC로 **반복·추론 시간** 부담(제약 7·8) → 주력 제외, **비교용**.
- **단순 GA(연관학습 없음)**: 큰 상호작용에서 building block을 깨 비효율 → 연관학습 EA 대비 열세(가설, Phase 2 검증).
- ➡️ 이 소거 과정이 곧 §1·§2 선택의 이유. **최종 우열은 초기 데이터로 벤치(research/09-A) 후 확정.**
