# 세미나 준비 로그

이 파일은 사용자가 tmp_ML 프로젝트를 이해한 바를 구술 → 검증하는 과정을 누적 기록한다.
형식: 각 세션마다 [사용자 발언 요약] → [검증자 피드백] 순으로 append.

검증자(나)의 역할: "논리를 깨는" 반박이 아니라, **알고 쓴 게 맞는지 확인하는 면접관**.
- 용어를 정확한 의미로 쓰고 있는가 (예: closure, effect sparsity, Chebyshev, OWA, Pareto 등)
- 결론의 근거(수치·실험조건)를 정확히 인용하는가
- 인과관계를 과장하거나 생략하지 않는가 (예: "블록 지식 덕" vs "그리디 좌표 효율 덕"의 구분)
- 일반화의 범위를 알고 있는가 (예: effect sparsity 가정이 깨지면 결론이 뒤집힌다는 전이조건)

---

## 참고: 프로젝트 배경 (검증자용 메모, 세미나 발표 자료 아님)

- **문제**: 30컬럼(categorical 10 + ordinal 20) 혼합 이산변수, 공간 크기 ~10^15,
  6개 목적(Y) 최적화. 실제 black-box 평가 1회=1분, 순차 평가만 가능, 예산 180/780/2400.
- **구조적 사전지식**: X가 common(10)/set1(5)/set2(15) 블록으로 나뉘고,
  set1은 y11,y12,y13에만, set2는 y21,y22,y23에만 영향(common을 통해서만 결합).
- **접근**: 실제 black-box 대신 정답을 아는 합성 벤치마크(BM1<BM2<BM3, functional-ANOVA 기반)를
  만들어 그 위에서 여러 옵티마이저(random/sobol/mlhs/block_coord_local/SA/GA/PSO/ACO/TPE/SMAC/BoTorch)를
  공정 비교.
- **핵심 결론**: 저예산(180~780) 영역에서는 "블록 구조를 활용한 그리디 좌표 local search"가
  BO/EA류를 이긴다. 이는 함수클래스가 effect-sparsity(주효과 지배)를 가질 때 나타나는 현상이며,
  이 가정이 깨지면(교호작용이 주효과를 압도하면) 결론이 전이되지 않는다.

## 1. 문제 정의 / 변수 특징 (사용자 발언, 원문 축약)

> tmp_ML은 매우 큰 binary X0 공간을 도메인 지식으로 categorical/ordinal로 줄여 10^15 수준의 X를
> 만든 문제. Y는 2 set, 각 set은 2D array 통계 2개 + 스칼라 1개 → "결국 스칼라량 3개".
> 입력공간(10^15) 대비 실제 생성 가능 데이터는 10^4 수준이라 데이터가 매우 부족.
> X: feature engineering 완료된 ordinal/categorical, "X 거리가 클수록 Y 변화도 크다"는 인터뷰 결과 있음.
> Y: 목표 6개이므로 Score 지표 필요. SPEC 없으면 space filling으로 Y분포를 먼저 수집해 정규화.
> score 지표는 필요시 가중치 조절 가능.

### 검증자 피드백
1. **[확인요망] "결국 스칼라량 3개"** — 앞 문장은 set당 3개(2D array 통계 2 + 스칼라 1)라고 했는데,
   set이 2개이므로 총 6개(y11~y13, y21~y23)여야 함. "3개"는 set 1개 기준 오기로 보임 — 실제 목적함수는 6개.
2. **[확인요망] "실제 생성 가능 10^4"** — 레포 예산은 180/780/2400(iteration), 2400도 ~2.4×10^3.
   10^4의 근거 재확인 필요(격차 방향은 맞으나 자릿수가 다름).
3. **[출처 불명] "binary X0 원공간"** — 00_Plan.md/AGENTS.md엔 이 원본 binary 공간 언급 없음.
   실제 인터뷰 배경지식이면 출처 명확히 하고, 세미나에서 얼마나 구체적으로 말할지(대외비 경계) 점검 필요.
4. **[좋은 포인트, 단서 필요] "X 거리 ↑ → Y 변화 ↑"** — 이 가정은 ordinal에만 자연스럽게 적용됨
   (순서가 있어야 "거리"가 의미 있음). categorical은 generator.py에서 레벨별 랜덤효과로 순서 없이 모델링.
   이 smoothness 가정이 block_coord_local의 1-hop 좌표탐색이 먹히는 이유와 직결 — 연결 지어 설명하면 좋음.
5. **[개념은 맞으나 설계와 다름] "space filling으로 Y분포 사전수집 후 normalize"** — 레포 실제 설계는
   별도 사전 단계가 아니라 "BO가 관측치로 온라인 적응 정규화"(AGENTS.md). 저예산(180)에서 정규화 목적
   space-filling에 eval을 쓰는 것도 예산 트레이드오프(탐색+확정 예산분할과 유사한 구조)임을 짚을 필요.
6. **[정확함]** score 가중치 조절 가능 — Chebyshev의 wᵢ, OWA의 k에 해당. 6목적 → 단일 스코어 필요성 논리도 정확.

## 2. Surrogate + 2D array 임베딩 (사용자 발언)

> surrogate model을 만든다면 2D array를 직접 임베딩하는 방법도 좋아 보임. 다만 저차원으로 녹여내야 하고
> 데이터 수가 많이 필요해 보이므로 후순위로 미뤄야 함.

### 검증자 피드백
- 방향 자체(2D array를 latent embedding으로 학습해 surrogate 입력에 쓰는 아이디어)는 합리적이지만,
  **현재 레포 구조와는 다른 접근**임을 짚고 넘어가야 함: 지금 파이프라인은 2D array를 이미 y_1, y_2라는
  **집계된 통계량(스칼라)**으로 요약한 뒤 그 스칼라들에 대해 최적화한다(generator.py는 2D array를 만들지도
  않고, 6개 스칼라 Y만 다룸). 즉 "2D array 임베딩"은 raw black-box(실제 calculator) 단계 얘기지,
  지금 벤치마크/알고리즘 비교 프레임워크(Phase 1~4) 안의 이야기가 아님. 이 구분을 명확히 하고 발표해야
  청중이 "왜 지금 안 하냐"고 물었을 때 헷갈리지 않음.
- **"데이터 수가 많이 필요해 후순위"** 판단은 정확함 — 예산이 180~2400인 상황에서 딥러닝 기반 임베딩을
  학습할 데이터가 절대적으로 부족(surrogate 자체도 GP 정도가 한계인 이유와 같은 맥락: botorch도 O(n³)
  제약으로 고예산에서 불리하다는 레포 결론과 같은 방향). 다만 "왜 부족한가"를 한 걸음 더 들어가면 좋음:
  임베딩은 보통 (a) 사전학습(다른 도메인 데이터로) 하거나 (b) 이 문제의 관측치만으로 학습하는데,
  (b)라면 2400개조차 2D array 구조를 일반화할 표현학습엔 턱없이 부족 — "몇 개면 될지"에 대한 감(예: 최소
  수천~수만 샘플 필요)까지 얘기하면 후순위 판단의 근거가 더 탄탄해짐.
- 참고로 이 아이디어가 실현 가능해지는 조건은 "2D array 자체에 접근 가능하고, calculator를 몇 번이고
  공짜로 돌릴 수 있을 때"인데, 지금 문제 정의(1 eval=1분, 순차평가, 예산 제한)와는 정면으로 배치되는
  전제라는 점도 인지하고 있는지 확인이 필요.

## 3. 피드백에 대한 사용자 답변 및 후속 논의

- **1,3,4번 정정 확인**: Y 6개 전체(set당 3×2set), 무효조합 제거된 X 공간, categorical엔 거리-변화
  상관 가정이 적용 안 됨을 인지 — 모두 정정 완료.
- **2번 재확인**: "10^4"는 단일 최적화 실행 예산(180/780/2400)이 아니라 **여러 번의 실행/운영 이력에
  걸쳐 장기간 누적되는 전체 데이터량**. 단일 캠페인 예산(~10^2~10^3)과 장기 누적치(~10^4)는 서로
  다른 스케일이므로 세미나에서 반드시 구분해서 말할 것. (→ 3번 임베딩 논의의 전제가 됨)
- **5번**: "알고리즘이 알아서 정규화" 방향은 레포 설계와 일치. 단, 적응적 정규화는 탐색 초반
  range 추정이 불안정(non-stationary normalization)하다는 캐치를 추가로 짚음 — 사용자는 아직
  이 완화책까지는 답변하지 않음(열린 질문으로 남음).
- **6번(2D array 임베딩)**: 사용자가 "future work라고만 하면 상위자가 좋아하지 않을 것 같다"며
  근거 보강을 요청. 검증자가 제시한 반박 논리 3가지:
  1. **스코프 위반** — y_m1,y_m2는 도메인 통계량으로 이미 정의됨(AGENTS.md: "벤치마크 목적함수·노이즈·
     scoring 정의 변경 금지"). 임베딩으로 대체하는 건 알고리즘 개선이 아니라 Y 정의 자체의 재설계 →
     현재 프로젝트(Phase 1~5) 범위 밖, 별도 연구 질문.
  2. **데이터/파라미터 비율** — 장기 누적 10^4조차 2D array의 픽셀/차원 수와 비슷하거나 못 미칠 수
     있어, "데이터 쌓이면 가능"이 아니라 "이 스케일에서 가능한지 자체가 불확실".
  3. **대체 근거 부재** — 기존 손통계량 대비 임베딩이 우월하다는 실증 없이 방향 전환을 제안하는 셈.
  → 권장 표현: "몰라서 미룬 것"이 아니라 "스코프·전제조건을 알고 판단해 제외한 것"으로 서술.

## 4. 5번(정규화) 심화 / 6번(임베딩) 구체화

### 5번 — 사용자 주장
알고리즘을 greedy/model기반/자연모사 3분류로 보고, "space-filling 표본의 분포가 크게 안 바뀐다"는
가정 하에 사전 정규화가 greedy·자연모사엔 괜찮고 model기반(BO)엔 안 좋아 보인다는 의견.

### 검증자 피드백
- 결론(모델기반이 상대적으로 불리)엔 동의하나, **근거는 재구성 필요**: "표본분포가 안정적"이라는
  가정 자체는 알고리즘 종류와 무관하게 동일한 리스크(effect-sparsity 하에서 진짜 최적은 희귀한
  극단치일 수 있어 소표본이 못 볼 위험은 모두에게 같음).
- 더 정확한 메커니즘: **"정규화가 필요한가"가 아니라 "정규화용 eval 소모의 기회비용이 얼마나 큰가"**.
  greedy(block_coord_local)는 원래 marginal-balanced 초기설계로 시작 → space-filling이 추가비용이
  아니라 재활용. GA/PSO/ACO도 초기 population이 랜덤/공간채움류라 마찬가지. 반면 BO는 "각 eval을
  최대한 정보가 되게 쓴다"가 존재 이유(FINDINGS.md의 sample-efficient BO) → 순수 정규화용 소모의
  기회비용이 상대적으로 더 큼.
- 부가: Chebyshev/OWA는 6목적 간 **상대적** 스케일 오차가 "어느 목적이 최악인가" 판단을 바꿔버리므로,
  정규화 오차의 악영향은 sum보다 Chebyshev/OWA에서 알고리즘 무관하게 더 큼 — 발표에 추가할 포인트.

### 6번 — 사용자 clarification
스코프 프레이밍("문제해결 제안")은 수용. 임베딩 의도는 **surrogate의 hidden layer에 (2D array를)
임베딩하는 것**이었다고 명확화. 필요 데이터량 산정을 위해 도움 요청.

### 검증자 피드백 (핵심 질문)
- 결정적 구조 문제: surrogate의 역할은 **아직 평가 안 한 X**에 대해 Y/score를 예측해 다음 후보를
  고르는 것인데, 2D array는 calculator를 **실제로 돌려야만** 나오는 중간 산출물 → 추론 시점에
  없는 값을 입력으로 쓸 수 없음.
- 두 갈래로 명확히 구분 필요:
  (a) 2D array가 입력측 임베딩 → X→2D array 매핑까지 별도로 학습해야 함(원 문제보다 어려운 2단계
      문제를 새로 만드는 셈, 데이터 요구량 오히려 증가).
  (b) 2D array가 학습시점 보조 타깃(auxiliary/multi-task loss)으로만 쓰이고, 추론 시엔 X→Y 직접
      예측 — 구조적으로 말이 됨(2D array가 추론 때 불필요).
- 사용자가 (a)/(b) 중 어느 쪽을 염두에 뒀는지 아직 답변 대기 중(열린 질문). 필요 데이터량 추정은
  이 선택에 따라 완전히 달라짐 → 방향 확정 후 이어서 계산 예정.

## 5. 차원축소 방법론 공백에 대한 답변

- 사용자가 "차원축소를 뭘로 할지 답을 못해 발표하기 어렵다"고 함.
- 검증자 제안: 답을 모르는 것 자체보다, **후보조차 없는 것**이 문제. 최소 PCA/SVD(선형) vs
  autoencoder(비선형) 두 후보를 놓고, 이 프로젝트의 핵심 교훈("저예산/저데이터에선 단순한 방법이
  이긴다" — FINDINGS.md의 block_coord_local > BO/GA 결론)과 같은 논리로 **선형(PCA류)을 우선
  검증 대상으로 제시**하면 "원칙에 따라 우선순위를 정한 것"으로 방어 가능. 참고: 현재 y_m1,y_m2도
  도메인 지식 기반 "손-PCA"에 가까운 baseline으로 이미 존재.

## 6. 문제정의/변수특징 재정리 (수정본) 및 피드백

### 사용자 수정본 요지
- X: binary·categorical·ordinal로 구성된 10^15 공간(무효조합 제거됨). Y: 6 스칼라(정정 반영).
  입력공간 10^15 vs 실제 데이터 10^4(장기 누적, 세션3에서 확정한 스케일 유지).
- X 거리-Y변화 상관은 "categorical은 확인 필요"로 유보 처리(정정 반영).
- Y: score 정규화 필요성 언급 + "iteration 돌면서 scaler가 업데이트돼야 하는가?"를 열린 질문으로 제시.
  가중치 조절 가능성 유지.

### 검증자 피드백
- **[분류 정정 필요] "binary, categorical 또는 ordinal"** — 3개 병렬 타입처럼 보이나, 실제로는
  **binary = categorical/ordinal의 특수케이스(cardinality=2)**. 00_Plan.md cardinality 배정에
  `2`가 common/set1/set2 전체에 섞여 있음(예: common `{30,10,5,4,3,3,2,2,2,2}`). 발표 시
  "categorical 10 + ordinal 20, 이 중 카디널리티 2인 것이 binary"로 표현할 것.
- **[좋은 습관]** categorical 거리가정에 "확인 필요" 플래그, 정규화를 단정 대신 질문으로 남긴 점.
- **[질문에 답변]** "scaler가 iteration마다 업데이트돼야 하는가" → 예(AGENTS.md: 실문제는 관측치로
  적응 정규화). 실무 원칙 2가지 제시: (1) **확장전용(expand-only)** 갱신(min/max는 넓히기만),
  (2) 과거 관측점의 정규화 점수도 **현재 스케일러 기준으로 재계산**해야 "지금까지의 best" 추적이
  일관됨(raw Y 보존해두면 추가 평가 없이 재계산 가능) — BoTorch류가 매 refit마다 하는 방식과 동일 원리.

## 7. 스케일러 변경의 부작용 심화 질문 + 2D array 문단 초안 검토

### 질문: "매 iter 스케일러가 바뀌면 다음 iter 계산에 문제 되지 않나?"
### 검증자 답변 요지
- **[문제 됨, 나이브하게 하면]** 두 실패모드: (1) **surrogate 라벨 드리프트**(model-based 한정) —
  과거/현재 스케일러로 계산된 score가 섞여 학습 데이터로 들어가면 X 불변인데 라벨이 흔들리는
  것처럼 보임 → 매 refit 전 **raw Y 전체를 현재 스케일러로 재정규화**(patch 아닌 재계산) 필요.
  (2) **incumbent(현재 best) 불안정** — EI 등 acquisition의 기준점이 스케일러 변화로 흔들려
  초반(관측치 적어 range 추정 변동 큼) 의사결정이 불안정. 데이터 누적되면 자연 수렴.
- **해법**: 확장전용 min/max + raw Y 보존 재계산 + (신규) **소규모 사전 space-filling으로 range를
  먼저 안정화하는 것도 BO에 정당화 가능** — 단, 5번 논의(정규화용 space-filling은 BO에 기회비용
  크다)와는 **다른 이유**(탐색 예산 절약 vs 초반 acquisition 안정화)이므로 구분해서 발표할 것.
- greedy(block_coord_local)는 상대적으로 강건: 매 비교가 "현재점 vs 이웃 1개"뿐이고 둘 다 현재
  스케일러로 재계산하면 sum은 전역 단조변환 불변 → 비교결과 안 바뀜(단 Chebyshev/OWA는 4번의
  상대스케일 문제 여전).

### 2D array 문단 초안 검토 (사용자가 세미나용으로 다듬은 문장)
> "surrogate 활용 시 array는 statistic으로 나타낼 수 없는 정보를 담고 있기 때문에 X→Array→Y 관계를
> 학습시키면 더 나은 학습 기대. 탐색공간이 커서 surrogate 구축 필요 데이터 대비 적재 가능 데이터
> 부족. array 차원축소 방식 미정."

### 검증자 피드백 (3곳 수정 권고)
1. "담고 있기 때문에" → **입증 아닌 가설**이므로 "담고 있을 것이라는 가설/기대"로 톤 낮출 것.
2. "X→Array→Y 관계를 학습" → 세션4의 (a)/(b) 미해결 질문이 재등장. 문장이 순차예측(a)로 읽히면
   "안 평가한 X의 Array를 어떻게 아느냐" 문제가 그대로 남음. 의도가 (b)(학습시 보조타깃, 추론은
   X→Y 직접)라면 그렇게 명시할 것.
3. "탐색공간이 커서 데이터 부족" → 원인 두 개가 섞임: (i) 일반적 예산부족(탐색공간 10^15 vs 예산,
   모든 방법에 공통) (ii) array-embedding 고유의 추가 데이터 요구(array 차원/네트워크 파라미터 대비
   샘플 수). "일반적 부족 위에 array 임베딩은 추가로 더 필요하다"는 이중구조로 쓸 것.

## 8. 2D array 문단 제외 결정 + 데이터 특징 4가지 검토

4. 각 컬럼이 블록화되어 있을 가능성 있음, 블록간 교호작용 무시 금지 →
   공통→set2(컬럼수多)→set1 순서 최적화가 유리할 가능성

### 검증자 피드백
1. **[정확, 용어 매칭]** = effect sparsity(00_Plan.md 용어). 발표 시 이 표준 용어 사용 권고.
2. **[좋은 flag, 마무리 필요]** 외부지식으로 잘 분리함. 방향성이 확인되면 단순 smoothness보다

   열린 질문처럼 안 보이게 할 것.
3. **[정확]**
4. **[수정 필요, 2곳]**
   - 3번과 모순: 블록 구조는 레포에서 "확정된 설계결정/실제 시스템 구조 반영"(given)이지 가능성이
     아님. "가능성이 있음" 표현 삭제하고 3번과 합쳐 단일 서술로.
   - 순서 근거 오류: "set2 컬럼수가 많아서"가 아니라 **set2의 유효차원이 커서(공통10+전용15=25컬럼,

     "병목부터 공략" 논리로 교체할 것. 실제 block_coord_local 순서(common→set2→set1)와 정확히 일치.

## 10. 2번 질문 구체화 + Score 체계 정리본 검토

### 질문 구체화: "column 14가 Y2 감소·Y5 증가시킨다는 도메인 지식을 모델/알고리즘에 반영해야 하는가?"
### 검증자 답변
반영해야 함. 이 예시 자체가 세션9의 "방향 엇갈림(=목적충돌)" 케이스 — 극값 고정이 아니라
score(스칼라라이제이션)에 맡길 대상. 반영 방식은 **하드 고정이 아니라 정보성 prior/초기화
편향**으로 넣을 것(교호작용에 따라 주효과 방향이 뒤집힐 위험 — FINDINGS.md 전이조건과 같은 경계).
방법론적으로는 이미 블록구조를 모든 알고리즘에 공정 주입(`_blk`)한 것과 같은 종류의 도메인
prior 주입이라 일관됨.

### Score 체계 정리본 검토
사용자가 전제조건 2개(정규화 필수+scaler 업데이트 이유 자기점검, 합기반 score의 위험) +
3개 지표(Aug. Chebyshev, OWA/bottom-k OWA, 단순가중합) 요약.

- **전제조건 2개**: 정확. 특히 "scaler 업데이트 이유를 설명할 수 있어야 한다"는 세션6~7과 일치.
- **Aug. Chebyshev**: 구조(최악 성분 부양 + 동점자 합으로 tie-break, max→min 부호변환)는 정확히
  유도됨. **계수 오류 1건**: 레포 정의(00_Plan.md §4)의 증강항 `ρ·Σ(zᵢ*−zᵢ)`은 **가중치 wi 없이**
  순수 합인데, 사용자 식은 `ρ·Σ(wi·si)`로 증강항에도 wi를 곱함 → 최악항·tie-break항 양쪽에 가중치가
  이중 작용해 의도한 균형이 미묘하게 바뀜. `min(wi·si) + ρ·Σ(si)`로 수정 필요.
- **smooth(softmin) 언급**: "X가 이산이라 연속 응답 불필요"는 방향은 맞으나 더 정확한 인과는
  "포트폴리오의 모든 알고리즘이 gradient 아닌 함수값만 쓰는 black-box 옵티마이저라 score의
  미분가능성 자체가 불필요"임 — X의 이산성은 애초에 gradient 방법을 쓸 이유가 없다는 것과 연결된
  결과이지 직접 인과는 아님.
- **OWA/bottom-k OWA, 단순가중합**: 둘 다 정확. "전제조건이 틀리면 다른 지표 필요"라는 메타 코멘트도 적절.

## 11. 후속 질문 2건 — 코드로 검증

### Q1: "타이브레이크(증강)항에 원래 가중치가 없나요?"
`benchmark/scoring.py:64`로 확인: `return -((w * gap).max(axis=-1) + rho * gap.sum(axis=-1))`.
`max`항엔 `w*gap`(가중치 적용), `rho*gap.sum(...)`항은 **가중치 없는 순수 합**. 문서(00_Plan.md)와
코드 일치. 세션10 지적("증강항에 wi 빼야 함") 확정.

### Q2: "gradient 기반 알고리즘이 있었다면 smooth가 필요했을까? 이산공간에서 gradient descent가
가능한가? 모델 기반이어야 하지 않나?"
- **사용자 직관이 정확함**: 이산공간 자체엔 gradient가 정의 안 되므로, gradient 기반 방법은 반드시
  (i) 연속완화(continuous relaxation) 또는 (ii) surrogate 모델(GP/NN 등 태생적으로 매끄러운 모델)을
  거쳐야 함.
- **실제로 이미 존재**: `botorch`(README: "GP-BO, qLogEI, 연속완화")가 정확히 이 케이스 — X를
  연속공간으로 relax하고 acquisition을 L-BFGS 등 gradient로 최적화 후 이산값으로 재투영.
- **세션10 답변 정정**: gradient가 걸리는 대상은 **raw score 수식(Chebyshev max/min)이 아니라
  GP가 학습한 surrogate**. GP posterior는 원함수의 매끄러움과 무관하게 커널 기반으로 항상 매끄러움
  → gradient 기반 알고리즘이 포트폴리오에 있어도(botorch) raw score의 smooth(softmin) 버전은
  여전히 불필요. 더 정확한 인과: "이 파이프라인의 모든 알고리즘은 raw score 수식을 직접 미분하지
  않고 항상 함수값(혹은 그 값을 학습한 매끄러운 surrogate)만으로 의사결정한다" — softmin이 필요해지는
  유일한 경우는 Chebyshev 공식 자체를 화이트박스로 직접 analytic 미분해 최적화하려 할 때뿐이며,
  포트폴리오엔 그런 알고리즘 없음.

## 12. 재질문 — "블랙박스라 미분불가능하니 Chebyshev를 미분불가능하게 써도 되는 것인가?"

### 검증자 답변 (핵심 확인 + 정밀화 2가지)
- **핵심 인과는 정확함**: X가 이산이든 연속이든 `calculator(X)→Y`가 블랙박스라 dY/dX 자체가
  존재하지 않음 → Chebyshev가 매끄러워도 뒤에 연결된 Y(X)에서 체인룰이 이미 끊김. "Chebyshev를
  미분가능하게" 만드는 것은 이 파이프라인 어디서도 무의미.
- **BoTorch와 모순 없음을 한 겹 더**: BoTorch의 gradient는 GP가 **관측된 score 숫자들**로부터
  학습한 surrogate에 걸림. GP는 각 X의 score가 어떤 공식(max/min이든 softmin이든)으로 계산됐는지
  모르고 그냥 **스칼라 라벨**로만 소비 → 생성 공식의 미분가능성 자체가 GP 입장에선 무의미.
- **추가 정밀화**: 설령 화이트박스+연속X를 가정해 raw Chebyshev를 직접 analytic 미분하려 해도,
  max/min은 **동점(tie, 측도 0)만 제외하면 거의 모든 곳에서 미분 가능**한 piecewise-linear
  함수(subgradient 항상 존재). softmin의 실제 역할은 "미분가능성 확보"가 아니라 **동점 근처
  꺾임(kink)이 gradient descent 수렴을 느리게 만드는 걸 완화하는 최적화 편의 장치**에 가까움.
- **한 문장 요약**: "이 파이프라인에선 아무도 raw score 수식을 analytic하게 미분하지 않는다 —
  블랙박스라 원천적으로 불가능하고(X 타입 무관), 화이트박스였어도 아무도 그렇게 안 한다
  (BoTorch조차 surrogate만 미분). 그래서 hard-min이든 soft-min이든 상관없다."

## 13. 재구성 질문 — "나중에 gradient descent 쓰는 알고리즘이면 달라지나? PSO는?"

### 사용자 프레이밍
GT 반응표면이 smooth인지 모르는 상태에서, X는 어차피 이산 선택. 처음엔 "X가 이산이라 score도
smoothing 불필요"라 생각했으나, 실은 "지금 쓰는 알고리즘이 gradient descent를 안 돌려서"라는
것으로 정정됨. 그럼 나중에 gradient descent 쓰는 알고리즘을 쓰면 달라지는가? PSO는 확률기반
근사기라 내부적으로 연속적인데 어떤가?

### 검증자 답변 (코드로 확인, `optim/algos.py`)
- **1층 (불변)**: 참 목적함수(calculator→Y)의 진짜 gradient는 블랙박스라 어떤 알고리즘을 써도
  여전히 못 구함.
- **2층 (진짜 갈림길, 아키텍처에 따라 달라짐)**: 코드 확인 결과 이 레포의 botorch는
  `SingleTaskGP(Xt, Ys)` — **Ys는 이미 Chebyshev/OWA/sum으로 스칼라화된 값**. 즉 "블랙박스 Y 관측
  → 스칼라화 → 숫자 하나에 단일 GP 학습 → acquisition gradient는 GP에만 걸림" 구조라 Chebyshev의
  내부 max/min 구조는 GP·acquisition 어디에도 노출되지 않음 → 이 레포에선 상관없음.
  **다른 설계였다면 다름**: 6개 Y 각각에 multi-output GP를 붙이고, acquisition 최적화 시점에
  "GP 예측값들 → Chebyshev(예측값) → 이 스칼라를 x로 미분" 하는 구조(다목적 BO 문헌의 ParEGO류)라면
  max의 꺾임(kink)이 실제로 gradient 기반 acquisition 최적화를 방해 → 이 경우엔 softmin이 진짜 도움.
  → "gradient 쓰는 알고리즘이면 달라지나"의 정확한 답: **어떤 지점에서 gradient를 거는지(스칼라화
  이후 단일 surrogate 미분 vs 스칼라화 이전 multi-output 예측값 위에서 미분)에 따라 다르다.**
- **PSO**: 내부 위치/속도 표현이 연속적인 것은 맞음(이 레포의 `run_pso_mixed`도 categorical엔
  logit+softmax 사용). 그러나 PSO는 **fitness 값(숫자)만 비교**해 개인최고/전체최고로 끌어당길 뿐
  score를 한 번도 미분하지 않는 0차(derivative-free) 방법 — GA/SA/ACO와 동일 계열. "탐색공간
  표현의 연속성"과 "score gradient 필요 여부"는 서로 다른 축이며, 진짜 기준은 "score 수식을 x에
  대해 실제로 미분하는가"임. PSO는 아니므로 smooth score 불필요 쪽에 남음.
