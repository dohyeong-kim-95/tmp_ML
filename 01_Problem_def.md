---

## 1. 한 줄 요약

> **N개(N>60)의 binary 입력 X로 결정되는 metric Y를 최대화하는 X 조합을, 제한된 실험 예산 안에서 찾는다.**
> 도메인지식 없이도 동작하는 일반적 baseline을 먼저 세우고, 이후 도메인지식을 주입해 발전시킨다.
> 예산 가정: **MAIN = 하루 320개(15주 누적 ~2.4~3.4만)**, SUB = 하루 40개(누적 ~수천). 체급에 따라 전략이 다르다([`02_Plan.md`](02_Plan.md)).

---

## 2. 문제 정의

| 항목 | 내용 |
|------|------|
| 입력 X | binary 변수 N개 (N > 60) |
| 출력 Y | **Shmoo array에서 rule-base로 파생되는 metric** (예: width / height / area 및 이들의 조합) — 구체 정의는 대외비 |
| 목표 | **Y를 최대화(또는 목표치 달성)하는 X 조합 탐색** |
| 탐색공간 | 최소 2^60 ≈ 1.15 × 10^18 (구조 제약으로 실제 가용 공간은 더 작음) |

### 변수 간 구조 (현재는 추상 정보만)
- 일부 binary는 **합쳐져 하나의 변량(그룹)** 을 의미함
- 일부 binary는 서로 **배타적(mutually exclusive)**
- 일부는 서로 **독립적**
- 일부는 **상호작용(interaction)이 매우 큼**
- ⚠️ 구체 규칙은 대외비. baseline은 이 구조를 **강하게 가정하지 않고**, 추후 제약/사전지식으로 **끼워넣을 수 있는 구조**로 설계한다.

### 문제 관점 (framing)
- 데이터 0건에서 시작하지만, **도메인 관점에서 X의 변화가 Y의 변화에 영향을 주는 "최적화 가능한 black-box 문제"** 로 본다.
- 즉 **문제의 최적화 가능성 자체는 도메인 지식으로 정당화**된다(무작위 잡음이 아니라 구조 있는 반응면이라는 전제).
- 초기 baseline은 **구체 도메인지식을 하드코딩하지 않는다.** 대신 구체 규칙은 추후
  **representation · constraint · prior · feature** 형태로 **점진 주입**한다(02_Plan §3 Phase 3, research/10).

## 3. 핵심 제약 (이 제약이 방법론을 결정한다 — 결정 논리는 02_Plan §1)

1. **기존 데이터 0건** — cold start. 초기 설계(initial design)부터 직접 만들어야 함.
2. **평가 비용/처리량 — 두 시나리오 (MAIN / SUB)**
   - 🟢 **MAIN(메인 가정): 40개/시간 × 8시간 = 하루 ~320개** → 15주 누적 **~2.4만~3.4만 = O(10^4)** (주5일 ~24,000, 주7일 ~33,600).
     - 정정: 이전 "O(10^5)"는 과장. 24시간 풀가동(960/일)·주7일이어도 ~10만으로 10^5 하한에 겨우 닿아 현실 가정은 **수만 = O(10^4)**.
   - 🔵 **SUB(서브 가정): 하루 ~40개** → 15주 누적 **O(10^3)**. 설비 가용시간이 줄거나 초기 워밍업 구간.
   - 두 경우 모두 **배치 평가**(MAIN은 시간당 40 micro-batch 또는 일 320 배치) → 배치 친화 알고리즘 필요.
3. **데이터 누적 가능** — 생산한 (X, Y)는 **계속 누적/관리** 됨. 매 반복마다 모델을 갱신하며 개선하는 반복 최적화 루프가 성립. (저장/버전관리 체계 필요)
4. **프로젝트 기간 = 15주 (고정)** — 총 실험 예산의 상한.
   - MAIN: 누적 **O(10^4)**(~2.4~3.4만) → "지도학습 대리모델 + 진화알고리즘"이 본령에 진입하는 체급.
   - SUB: 누적 **O(10^3)** → 표본효율이 지배하는 체급(BO 계열).
   - "초기 screening/워밍업 → 주력 루프 → 후반 도메인지식 주입"으로 **예산 배분 계획** 필요.
5. **예산 << 탐색공간** — MAIN의 O(10^4)도 2^60의 약 10^-14 → 전수 불가. **구조 활용**이 본질.
6. **구조 규칙은 추후 주입** — baseline 단계에서는 데이터로 구조(상호작용 등)를 *발견*하는 능력이 바람직.
7. **반복(iteration) 계산 시간 제약** — 매 배치 제안 시 **모델 적합 + 후보 탐색**이 과도하게 오래 걸리면 안 됨(특히 MAIN은 하루 320 배치를 매일 생성). → **데이터에 선형 확장되는 모델(트리/선형/진화모델)** 선호, GP-BO의 O(n³)/GP-HMC는 부담.
8. **추론(다음 후보 생성) 지연 한도 — 하드 제약** — **학습(training)은 시간 무제한**이지만, **추론(다음 60-bit 신호 1건 생성)은 ≤ 3초, 이상적으로 ≤ 500ms.** → 무거운 탐색은 **학습/오프라인으로 amortize**, 후보 방출은 즉시. (해결책은 02_Plan §5)
9. **측정 노이즈 존재** — 같은 X를 두 번 측정해도 Y가 다를 수 있음(크기는 미상). → 노이즈·선택편향 고려 필수(성공 기준 §6, 대응은 02_Plan).

## 5. 성공 기준

### 5.1 탐색 objective vs 프로젝트 성공 기준 (구분)
- **Optimizer의 탐색 objective = Y 최대화(maximize Y).** (탐색은 단순·명확한 목적함수로 구동)
- **프로젝트 성공 기준 = 내부 target threshold(Y_target)를 만족하는 X를 예산 내에 발견.** (satisficing/feasibility)
- 따라서 평가/기록은 best-Y만이 아니라 다음을 **함께 기록**한다:
  - **budget-to-target**: target 도달까지 쓴 실험 수
  - **threshold-hit count**: target 통과 후보 개수
  - **threshold-hit 후보의 다양성**: 통과 X들이 서로 얼마나 다른가
  - (보조) best-Y 곡선

### 5.2 측정·판정 정책 (노이즈 §3-9 대응)
- **최적화 루프 중에는 예산 효율을 위해 각 X의 관측 Y를 deterministic single observation처럼 사용**한다(반복측정 안 함).
- **단, Y_target을 통과한 후보는 별도 confirmation 단계에서 반복 측정**하고,
  **Monte Carlo / bootstrap / one-sided confidence interval / guardband 중 하나의 통계 기준**으로 **최종 성공 여부를 판정**한다.
  - 결정규칙은 *"관측 평균이 넘었다"가 아니라 "참값이 신뢰수준으로 Y_target을 넘는다"* — baseline 권고는 **one-sided LCB ≥ Y_target (=guardband, 95%)** + 다중성 보정. (research/09-G2)
- 근거: 단발 관측만으로 통과 선언 시 **winner's / optimizer's curse**(노이즈로 과대추정 → 재측정 시 평균 회귀). 2단계(discovery→confirmation)로 방어. (research/09-G)

### 5.3 보조·확장 기준
- 보조(sanity): 동일 예산에서 **무작위 탐색 대비 우수**, best-Y 곡선 우상향.
- 확장성: 도메인지식 주입 시 target 도달이 더 빨라지는가.

> ⚠️ Y_target의 **정량 값**, confirmation의 **반복 횟수·통계 임계**는 대외비/노이즈 의존 → §6. 정해져야 지표 확정 가능.
