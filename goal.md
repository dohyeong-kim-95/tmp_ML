# goal.md — Shmoo Score 최대화 Baseline 프로젝트

> 작성일: 2026-06-22
> 본 문서는 인터뷰를 통해 정리한 **목표/제약/접근 방향 명세서**입니다.
> 대외비(변수의 물리적 의미, 구조 규칙, 실제 score 수식 등)는 포함하지 않습니다.

---

## 1. 한 줄 요약

> **N개(N>60)의 binary 입력 X로 결정되는 score Y를 최대화하는 X 조합을, 제한된 실험 예산 안에서 찾는다.** 도메인지식 없이도 동작하는 일반적 baseline을 먼저 세우고, 이후 도메인지식을 주입해 발전시킨다.
> 예산 가정: **MAIN = 하루 320개(15주 누적 ~2~3만)**, SUB = 하루 40개(누적 ~수천). 체급에 따라 주력 방법이 다르다(§5).

---

## 2. 문제 정의

| 항목 | 내용 |
|------|------|
| 입력 X | binary 변수 N개 (N > 60) |
| 출력 Y | **Shmoo array에서 rule-base로 파생되는 metric** (예: width / height / area 및 이들의 조합) — 구체 정의는 대외비 |
| 목표 | **Y를 최대화하는 X 조합 탐색** (optimization 중심) |
| 탐색공간 | 최소 2^60 ≈ 1.15 × 10^18 (구조 제약으로 실제 가용 공간은 더 작음) |

### 변수 간 구조 (현재는 추상 정보만)
- 일부 binary는 **합쳐져 하나의 변량(그룹)** 을 의미함
- 일부 binary는 서로 **배타적(mutually exclusive)**
- 일부는 서로 **독립적**
- 일부는 **상호작용(interaction)이 매우 큼**
- ⚠️ 구체적 규칙은 대외비. baseline은 이 구조를 **강하게 가정하지 않고**, 추후 제약/사전지식으로 **끼워넣을 수 있는 구조**로 설계한다.

## 3. 핵심 제약 (이 제약이 방법론을 결정한다)

1. **기존 데이터 0건** — cold start. 초기 설계(initial design)부터 직접 만들어야 함.
2. **평가 비용/처리량 — 두 시나리오 (MAIN / SUB)**
   - 🟢 **MAIN(메인 가정): 40개/시간 × 8시간 = 하루 ~320개** → 15주 누적 **~2.4만~3.4만 = O(10^4)** (주5일 ~24,000, 주7일 ~33,600).
     - 정정: 이전 문서의 "O(10^5)"는 과장. **24시간 풀가동(960/일)·주7일이어도 ~10만으로 10^5 하한에 겨우 닿는 수준**이라 현실적 가정은 **수만 = O(10^4)**.
   - 🔵 **SUB(서브 가정): 하루 ~40개** → 15주 누적 **O(10^3)**. 설비 가용시간이 줄거나 초기 워밍업 구간에 해당.
   - 두 경우 모두 **배치 평가**(MAIN은 시간당 40 micro-batch 또는 일 320 배치) → 배치 친화 알고리즘 필요.
3. **데이터 누적 가능** — 생산한 (X, Y)는 **계속 누적/관리** 됨. 매 반복마다 모델(대리모델/연관모델)을 갱신하며
   개선하는 반복 최적화 루프가 성립한다. (데이터 저장/버전관리 체계 필요)
4. **프로젝트 기간 = 15주 (고정)** — 총 실험 예산의 상한을 결정.
   - MAIN: 누적 **O(10^4)**(~2.4~3.4만) 규모 → "지도학습 대리모델 + 진화알고리즘"이 본령에 진입하는 체급.
   - SUB: 누적 **O(10^3)** → 표본효율이 지배하는 체급(BO 계열).
   - "초기 screening/워밍업 며칠 → 주력 루프 → 후반 도메인지식 주입"으로 **예산 배분 계획** 필요.
5. **예산 << 탐색공간 (여전히 성립)** — MAIN의 O(10^4)도 2^60의 약 10^-14에 불과 → 전수 불가.
   - SUB(O(10^3)): **표본 효율**이 거의 전부 → BO 계열.
   - MAIN(O(10^4)): **표본효율 + 탐색 알고리즘 효율의 균형**, 그리고 **변수 구조(linkage/building block) 활용**이 핵심.
6. **구조 규칙은 추후 주입** — baseline 단계에서는 데이터로 구조(상호작용 등)를 *발견*하는 능력이 바람직.
7. **반복(iteration) 계산 시간 제약** — 알고리즘 자체의 wall-clock도 중요. 매 배치 제안 시
   **모델 적합 + 후보 탐색/획득함수 최적화**가 과도하게 오래 걸리면 안 됨(실험 주기·운영성 저해).
   특히 MAIN에서는 **하루 320개 배치를 매일** 생성하므로 모델 갱신·후보 생성이 빨라야 함.
   → N>60·데이터 누적 시 비용이 폭발하는 방식(GP-BO의 O(n³), GP-HMC 등)은 부담 →
   **데이터에 선형적으로 확장되는 모델(트리/선형/진화모델)** 을 선호.
8. **추론(다음 후보 생성) 지연 한도 — 하드 제약** — **학습(training)은 시간 무제한**이지만,
   **추론(다음 iteration의 60-bit on/off 신호 1건을 생성)은 ≤ 3초, 이상적으로 ≤ 500ms.**
   - 즉 "무거운 탐색"은 **학습/오프라인으로 amortize**하고, **후보 방출은 즉시**여야 함.
   - 설계 함의(§5.4): ① **후보 버퍼/큐**(오프라인 최적화로 미리 채우고 서빙은 dequeue, ~ms — 방법 무관)
     또는 ② **amortized 추론 모델**(학습된 정책/대리모델이 단일 forward pass로 후보 생성, 보고된 1~4ms)
     또는 ③ **anytime/예산제한 지역탐색**(500ms에서 중단, best-so-far 반환).
   - 이 제약은 BO·EA를 배제하지 않음 — 무거운 부분을 오프라인으로 옮기면 충족. 단 "요청 시점에
     매번 무거운 acquisition/탐색을 도는" 순진한 구현은 부적합.

## 5. 접근 방향 (권고 baseline)

> 핵심 인식: 이것은 **"고차원(N>60) binary 조합에 대한 비싼 black-box 최적화"** 문제이며,
> **실험 예산의 체급(regime)에 따라 최적 방법이 달라진다.**
> 본 프로젝트는 **MAIN = 하루 320개(누적 O(10^4))**, **SUB = 하루 40개(누적 O(10^3))** 를 가정한다.

### 5.0 체급별 주력 방법
| Regime | 누적 예산 | 지배 요인 | **주력 방법(현 가설 — Phase 2에서 검증 대상)** |
|--------|-----------|-----------|----------------|
| 🟢 **MAIN** (320/일) | O(10^4) | 구조 활용 + 알고리즘 효율 | **모델기반 연관학습 진화알고리즘(DSMGA-II/LT-GOMEA) + 지도학습 대리모델(SAEA)** |
| 🔵 **SUB** (40/일) | O(10^3) | 표본 효율 | **베이지안 최적화(BOCS형 희소회귀 / SMAC3 / TPE) + screening DoE** |

> 두 체급의 **공통 인프라**(인코딩/제약 슬롯, 데이터 스키마, 시뮬레이터, best-so-far 모니터링)는 하나로 두고,
> **옵티마이저만 교체**하는 구조로 설계한다(전환·비교 용이).

### 5.1 단계별 파이프라인

**Phase 0 — 인코딩 & 제약 표현 (도메인지식 슬롯, 공통)**
- binary → 모델 입력 표현으로 매핑. 배타/그룹은 **범주형 재파라미터화**, 조건부는 조건부 공간으로.
- 구조 규칙은 가능해지는 대로 제약/합성변수/사전지식으로 주입할 **슬롯만 비워둔다**.

**Phase 1 — 초기 데이터 확보 (cold start)**
- ⚠️ **DoE는 raw X0(60bit)가 아니라 인코딩된 feature space X에서 수행한다.**
  group/mutex/conditional이 있는 raw 비트는 "독립 2수준 요인" 가정을 깨므로, **먼저 X0→X 인코딩**(그룹→범주형,
  배타→범주형, 조건부→계층) 후 **합법 요인공간 X 위에서 설계**. (research/03, research/10)
- SUB/초기: X 위에서 **screening 설계**. 단 제약이 있으면 PB/분수계획(직교설계)을 그대로 쓰지 말고
  **제약-aware 최적설계(D-optimal, coordinate-exchange)** 로 — 직교설계에 repair를 가하면 균형/별칭구조가 깨짐.
- MAIN: 첫 며칠은 (제약-aware) DoE + 광역 무작위로 **수천 표본**을 빠르게 모아 지도학습 대리모델의 토대를 만든다.
- 세부 설계질문(candidate set·균형성·screening 범위·예산)은 research/03 §4A–4D 참조.

**Phase 2 — 주력 최적화 루프 (체급별)**

- 🟢 **MAIN — 모델기반 진화 + 대리모델 보조 (SAEA)**
  - **연관학습 EA(DSMGA-II / LT-GOMEA)** 로 변수 묶음(building block) 단위 탐색
    → binary 구조·큰 상호작용에 부합. **학습된 linkage는 구조 *추정치(가설)*** 이며 진위는 검증 대상(research/09-C).
  - **지도학습 대리모델(GBM/RF)** 으로 EA가 생성한 다수 후보를 **사전선별(triage)** → 상위 320개만 실측.
    이때 **순수 top-k가 아니라 다양성(DPP/클러스터/ε-탐색)을 함께** 고려해 배치 다양성 확보(research/09-D).
  - 매일: 후보 생성 → 대리모델 선별 → 320개 실측 → 데이터 누적 → 모델·linkage 갱신.

- 🔵 **SUB — 배치 베이지안 최적화**
  - 대리모델: (해석형) **BOCS형 희소 pairwise 베이지안 회귀** / (강건형) **SMAC3(RF)·Optuna(TPE)**.
  - 획득함수: EI / **배치 Thompson sampling**(q=40에 적합). q-EI는 q>4에서 비현실적.
  - 매일: 40점 제안 → 측정 → 모델 갱신 → 반복.

**Phase 3 — 도메인지식 주입 (발전 단계, 공통)**
- 구조 규칙을 제약/사전분포(prior)/커스텀 feature/커널/EA 연관모델 초기화로 주입.
- effect heredity 등으로 상호작용/연관 탐색을 가지치기.

### 5.2 후보 도구 스택 (네트워크/OSS 사용 가능 전제)
| 용도 | 가벼운 baseline | 본격 |
|------|----------------|------|
| MAIN 옵티마이저 | GA(`DEAP`) + 휴리스틱 linkage | **DSMGA-II / LT-GOMEA(GOMEA)** 공개 구현, hBOA |
| MAIN 대리모델 | `scikit-learn` RF | **XGBoost/LightGBM** + SHAP 해석 |
| SUB BO | Optuna (TPE, ask/tell 배치) | SMAC3(RF), BoTorch/Ax, BOCS(`BOCSpy`) |
| DoE | pyDOE2 | 커스텀 D-optimal |

> **추천 출발점(MAIN)**: DoE/광역 시드 → **연관학습 EA + GBM 대리모델 선별**.
> 이유: (1) O(10^4) 예산에서 진화탐색이 본령, (2) binary 구조(building block)를 *학습*해 활용·해석, (3) 하루 320 배치와 자연스럽게 부합, (4) 트리/진화모델은 데이터에 선형 확장 → 반복시간 통제.

> ⚠️ **반복 시간 가드레일**: 매일 배치(MAIN 320, SUB 40) 생성의 wall-clock을 점검. 목표(잠정): **배치당 수 분 내**.
> GP-HMC(SAASBO)·대형 GP-BO 등 n³/HMC 비용 방법은 N>60·누적 시 부담 → 주력 제외, 비교용으로만.

### 5.3 서빙 아키텍처 — 학습/추론 분리 (추론 ≤ 3s, 목표 ≤ 500ms)
> 핵심: **무거운 탐색 = 학습(오프라인, 무제한)**, **다음 60-bit 신호 방출 = 추론(온라인, 하드 한도)** 로 분리.

- **패턴 A — 후보 버퍼/큐 (1순위, 방법 무관, 가장 단순·견고)**
  - 오프라인에서 옵티마이저(EA/BO/SAEA)를 충분히 돌려 **다음 평가할 후보 N개를 미리 생성·정렬**해 큐에 적재.
  - 서빙은 큐에서 **pop만** → ~ms. MAIN은 어차피 하루 320 배치를 생성하므로 자연스럽게 부합.
  - 데이터가 누적되면 백그라운드에서 큐를 **재생성/갱신**.
- **패턴 B — amortized 추론 모델 (요청 시점 상태반영 생성이 필요할 때)**
  - 학습된 **정책/대리모델**이 현재 상태(history)를 받아 **단일 forward pass로 다음 후보**를 생성(보고 사례 1~4ms).
  - 무거운 최적화를 학습에 amortize → 추론은 feed-forward. (amortized optimization/BO 문헌)
- **패턴 C — anytime/예산제한 지역탐색**
  - GBM 대리모델 위에서 **500ms 예산**의 지역탐색(비트플립/타부서치)을 돌려 시간초 시 best-so-far 반환.
  - 대리모델 추론 자체는 ms 단위라 예산 내 다수 후보 평가 가능.

> 권고: **MAIN/SUB 공통으로 패턴 A(후보 큐)** 를 기본 채택(구현 단순·방법 무관·견고).
> 상태반영 즉시생성이 필요하면 패턴 B(amortized) 또는 C(anytime)를 추가.
> 어느 경우든 **무거운 옵티마이저는 학습 단계로** 밀어 넣어 추론 한도(≤3s, 목표≤500ms)를 보장.

### 5.4 설득 포인트
- **SUB**: 데이터 0건+점당 비용 큼 → 대량 라벨 지도학습/그리드 불가 → 능동적 BO가 본질적으로 맞음.
- **MAIN**: 예산이 10~30배 → **지도학습 대리모델과 진화탐색이 가능해짐**. 그러나 2^60은 여전히 못 훑으므로 "무작정 진화"가 아니라 **구조(linkage)를 학습·활용하는 모델기반 EA**가 정답.
- 두 체급 공통: **구조를 발견·활용**하는 모델이 추후 도메인지식 주입과 양방향으로 맞물림.

## 7. 성공 기준

- 🎯 **1차(주 기준): Y가 내부 기준(target threshold)을 통과하는 X 조합을 예산 내에 발견.**
  - 즉 순수 최대화가 아니라 **목표 달성(satisficing/feasibility)** 문제로 본다 — "충분히 좋은 X를 찾았는가".
  - 함의: best-Y 추적뿐 아니라 **threshold 도달까지의 실험 수(time/budget-to-target)**, **threshold 통과 X의 개수·다양성**도 지표가 됨.
  - threshold 통과 영역을 노리는 **level-set/feasibility 지향 탐색**(목표 근처를 집중 샘플)이 순수 최대화보다 유리할 수 있음(추후 검토).
- 보조 기준(sanity): 동일 예산에서 **무작위 탐색 대비 우수**(하한선 확인용), best-Y 곡선 우상향.
- 확장성: 도메인지식 주입 시 target 도달이 더 빨라지는가.

> ⚠️ threshold의 **정량 값**은 대외비/미정 → §8 미결 항목. 기준선(threshold)이 정해져야 위 지표를 확정 가능.
