# goal.md — Shmoo Score 최대화 Baseline 프로젝트

> 작성일: 2026-06-22
> 본 문서는 인터뷰를 통해 정리한 **목표/제약/접근 방향 명세서**입니다.
> 대외비(변수의 물리적 의미, 구조 규칙, 실제 score 수식 등)는 포함하지 않습니다.

---

## 1. 한 줄 요약

> **N개(N>60)의 binary 입력 X로 결정되는 score Y를 최대화하는 X 조합을, 극히 제한된 실험 예산 안에서 찾는다.** 도메인지식 없이도 동작하는 일반적 baseline을 먼저 세우고, 이후 도메인지식을 주입해 발전시킨다.

---

## 2. 문제 정의

| 항목 | 내용 |
|------|------|
| 입력 X | binary 변수 N개 (N > 60) |
| 출력 Y | shmoo plot의 width / height / area를 사칙연산으로 계산한 단일 score (대외비) |
| 목표 | **Y를 최대화하는 X 조합 탐색** (optimization 중심) |
| 탐색공간 | 최소 2^60 ≈ 1.15 × 10^18 (구조 제약으로 실제 가용 공간은 더 작음) |

### 변수 간 구조 (현재는 추상 정보만)
- 일부 binary는 **합쳐져 하나의 변량(그룹)** 을 의미함
- 일부 binary는 서로 **배타적(mutually exclusive)**
- 일부는 서로 **독립적**
- 일부는 **상호작용(interaction)이 매우 큼**
- ⚠️ 구체적 규칙은 대외비. baseline은 이 구조를 **강하게 가정하지 않고**, 추후 제약/사전지식으로 **끼워넣을 수 있는 구조**로 설계한다.

---

## 3. 핵심 제약 (이 제약이 방법론을 결정한다)

1. **기존 데이터 0건** — cold start. 초기 설계(initial design)부터 직접 만들어야 함.
2. **평가가 매우 비쌈** — Y 한 점을 얻으려면 생산설비를 써야 하고, 설비 가동 시간이 부족함.
   - 실효 처리량: **하루 ~40개** (제작 자체는 빠르나 설비 가용 시간이 병목)
   - 즉, **배치(batch) 단위로 하루 40개씩** 평가 가능 → 순차+배치 실험 설계가 자연스러움
3. **데이터 누적 가능** — 생산한 (X, Y)는 **계속 누적/관리** 됨. 매 반복마다 대리모델을 갱신하며
   개선하는 정통 순차 베이지안 최적화 루프가 성립한다. (데이터 저장/버전관리 체계 필요)
4. **프로젝트 기간 = 15주 (고정)** — 이것이 총 실험 예산의 상한을 결정.
   - 거친 추정: 40/day × 가동일 → 15주 동안 **누적 O(10^3) 표본** 규모 (가동일수는 미확정)
   - 따라서 "초기 screening에 며칠 / 이후 BO 루프에 나머지"로 **예산 배분 계획**이 필요.
5. **예산 << 탐색공간** — 누적 O(10^3) 표본 vs 10^18 공간. 무작위/전수는 불가능.
   → **표본 효율(sample efficiency)** 이 가장 중요한 성능 지표.
6. **구조 규칙은 추후 주입** — baseline 단계에서는 데이터로 구조(상호작용 등)를 *발견*하는 능력이 바람직.
7. **반복(iteration) 계산 시간 제약** — 알고리즘 자체의 wall-clock도 중요. 매 배치(40점) 제안 시
   **대리모델 적합 + 획득함수 최적화**가 과도하게 오래 걸리면 안 됨(실험 주기와 운영성을 해침).
   → N>60에서 비용이 폭발하는 방식(GP-BO의 O(n³), pairwise 항 SA 등)은 표본효율과 **계산비용을 함께**
   저울질해야 함. "표본효율 최고지만 한 배치 제안에 수 시간"인 방법은 baseline으로 부적절.

## 4. 1차 목표 (baseline)

- [ ] **최적 X 조합 탐색 파이프라인** 구축 (데이터 0건 → 반복 실험으로 Y 최대화)
- [ ] 도메인지식 없이 동작하지만, 나중에 제약/사전지식을 주입할 수 있는 **확장 가능한 구조**
- [ ] **"왜 이 방법이 baseline으로 적절한가"** 를 설득하는 산출물

### 비목표 (이번 단계에서 하지 않는 것)
- 구체적 도메인 규칙 하드코딩 (추후 단계)
- 완벽한 해석 모델 — 단, 상호작용 단서를 주는 정도의 해석성은 보너스로 추구
- 실시간/온라인 서빙

## 5. 접근 방향 (권고 baseline)

> 핵심 인식: 이것은 **"비싼 black-box 함수에 대한 고차원 조합 최적화 (expensive black-box combinatorial optimization)"** 문제다.
> 데이터 무한 + 평가 저렴이면 일반 지도학습/그리드서치로 충분하지만, **본 문제는 정반대**이므로
> **순차적 실험 설계(Design of Experiments) + 대리모델 기반 베이지안 최적화(Bayesian Optimization)** 계열이 baseline으로 가장 타당하다.

### 5.1 단계별 파이프라인 (제안)

**Phase 0 — 인코딩 & 제약 표현 (도메인지식 슬롯)**
- binary → 모델 입력 표현으로 매핑
- 그룹/배타 관계는 가능해지는 대로 제약(constraint) 또는 합성변수(composite)로 주입할 **슬롯만 비워둔다**
- baseline에서는 제약 없이 시작 가능

**Phase 1 — 초기 설계 (cold start, screening)**
- 무작위가 아니라 **선별 실험계획법(screening DoE)** 으로 첫 배치 구성
  - 예: fractional factorial / Plackett–Burman / D-optimal design
- 목적: 적은 표본으로 **주효과(main effect) + 저차 상호작용**의 신호를 최대한 확보
- 첫 1~2일치 배치(≈40~80점)에 해당

**Phase 2 — 대리모델 + 획득함수 반복 루프 (batch BO)**
- **대리모델(surrogate)** 후보:
  - (해석형) **희소 상호작용 선형/베이지안 회귀** — pairwise 항 포함, Lasso/Bayesian linear regression
    → 상호작용·변수중요도를 부산물로 제공해 추후 도메인지식화에 유리 (BOCS 계열)
  - (강건형) **랜덤포레스트/GBM 대리모델** — 범주/조건부 변수에 강함 (SMAC 계열)
  - (커널형) binary용 커널 GP — 예: Hamming/diffusion kernel (COMBO 계열)
- **획득함수(acquisition)**: Expected Improvement / Thompson Sampling 등
- **배치 제안(batch proposal)**: 하루 40점을 한 번에 뽑아야 하므로 **q-batch (q-EI, batch Thompson)** 적용
- 매일: 40점 제안 → 측정 → 모델 업데이트 → 반복

**Phase 3 — 도메인지식 주입 (발전 단계)**
- 구조 규칙을 제약/사전분포(prior)/커스텀 feature/커널로 주입
- 상속 원리(effect heredity) 등으로 상호작용 탐색을 가지치기

### 5.2 후보 도구 스택 (네트워크/OSS 사용 가능 전제)
| 용도 | 가벼운 baseline | 본격 |
|------|----------------|------|
| BO 프레임워크 | Optuna (TPE, ask/tell 배치) | BoTorch / Ax (batch GP-BO) |
| 범주·조건부 강건 | — | SMAC3 (RF 대리모델) |
| 조합 BO 특화 | 희소 선형회귀 + 시뮬레이티드 어닐링 (BOCS 직접 구현) | COMBO |
| DoE | pyDOE 계열 | 커스텀 D-optimal |

> **추천 출발점**: Phase1 screening DoE + **희소 pairwise 선형/베이지안 회귀 대리모델** + batch acquisition.
> 이유: (1) 비싼 평가에 맞는 표본 효율, (2) binary/조합 입력 native 처리, (3) 상호작용을 *발견*해 추후 도메인지식화의 다리가 됨, (4) 하루 40점 배치와 자연스럽게 맞음, (5) 선형 대리모델은 적합이 가볍고 획득함수 최적화도 정수계획/SA로 **반복 시간이 통제 가능**.

> ⚠️ **반복 시간 가드레일**: 후보 방법은 표본효율뿐 아니라 **한 배치(40점) 제안에 걸리는 wall-clock**으로도 평가한다.
> 목표 가이드(잠정): 한 배치 제안이 **수 분 내**. GP-BO/COMBO처럼 n 증가에 민감한 방법은 N>60·누적표본 증가 시 비용 점검 필수.

### 5.3 왜 단순 지도학습/그리드서치가 baseline으로 부적절한가 (설득 포인트)
- 데이터 0건 + 점당 비용 큼 → 대량 라벨 전제의 지도학습은 성립 안 함
- 2^60 공간에서 그리드/랜덤 서치는 표본 효율이 너무 낮음
- 따라서 "**평가 예산을 어디에 쓸지 능동적으로 정하는**" active/Bayesian 계열이 본질적으로 맞음
