# goal.md — Shmoo Score 최대화 Baseline 프로젝트

> 작성일: 2026-06-22
> 본 문서는 인터뷰를 통해 정리한 **목표/제약/접근 방향 명세서**입니다.
> 대외비(변수의 물리적 의미, 구조 규칙, 실제 score 수식 등)는 포함하지 않습니다.

---

## 1. 한 줄 요약

> **N개(N>60)의 binary 입력 X로 결정되는 score Y를 최대화하는 X 조합을, 제한된 실험 예산 안에서 찾는다.** 도메인지식 없이도 동작하는 일반적 baseline을 먼저 세우고, 이후 도메인지식을 주입해 발전시킨다.
> 예산 가정: **MAIN = 하루 320개(15주 누적 ~2~3만)**, SUB = 하루 40개(누적 ~수천). 체급에 따라 주력 방법이 다르다(§5).

---

## 2. 문제 정의

| 항목 | 내용 |
|------|------|
| 입력 X | binary 변수 N개 (N > 60) |
| 출력 Y | shmoo plot의 width / height / area를 사칙연산으로 계산한 단일 score (대외비) |
| 목표 | **Y를 최대화하는 X 조합 탐색** (optimization 중심) |
| 탐색공간 | 최소 2^60 ≈ 1.15 × 10^18 (구조 제약으로 실제 가용 공간은 더 작음) |

### 변수 간 구조 (현재는 추상 정보만)
- 일부 binary는 **합쳐져 하나의 변량(그룹)** 을 의미함
- 일부 binary는 서로 **배타적(mutually exclusive)**
- 일부는 서로 **독립적**
- 일부는 **상호작용(interaction)이 매우 큼**
- ⚠️ 구체적 규칙은 대외비. baseline은 이 구조를 **강하게 가정하지 않고**, 추후 제약/사전지식으로 **끼워넣을 수 있는 구조**로 설계한다.

## 3. 핵심 제약 (이 제약이 방법론을 결정한다)

1. **기존 데이터 0건** — cold start. 초기 설계(initial design)부터 직접 만들어야 함.
2. **평가 비용/처리량 — 두 시나리오 (MAIN / SUB)**
   - 🟢 **MAIN(확정 가정): 40개/시간 × 8시간 = 하루 ~320개** → 15주 누적 **O(10^4~10^5)** (주5일 ~24,000, 주7일 ~33,600).
   - 🔵 **SUB(보조 가정): 하루 ~40개** → 15주 누적 **O(10^3)**. 설비 가용시간이 줄거나 초기 워밍업 구간에 해당.
   - 두 경우 모두 **배치 평가**(MAIN은 시간당 40 micro-batch 또는 일 320 배치) → 배치 친화 알고리즘 필요.
3. **데이터 누적 가능** — 생산한 (X, Y)는 **계속 누적/관리** 됨. 매 반복마다 모델(대리모델/연관모델)을 갱신하며
   개선하는 반복 최적화 루프가 성립한다. (데이터 저장/버전관리 체계 필요)
4. **프로젝트 기간 = 15주 (고정)** — 총 실험 예산의 상한을 결정.
   - MAIN: 누적 **O(10^4)**(~2~3만) 규모 → "지도학습 대리모델 + 진화알고리즘"이 본령에 진입하는 체급.
   - SUB: 누적 **O(10^3)** → 표본효율이 지배하는 체급(BO 계열).
   - "초기 screening/워밍업 며칠 → 주력 루프 → 후반 도메인지식 주입"으로 **예산 배분 계획** 필요.
5. **예산 << 탐색공간 (여전히 성립)** — MAIN의 O(10^4)도 10^18의 10^-13에 불과 → 전수 불가.
   - SUB(O(10^3)): **표본 효율**이 거의 전부 → BO 계열.
   - MAIN(O(10^4~5)): **표본효율 + 탐색 알고리즘 효율의 균형**, 그리고 **변수 구조(linkage/building block) 활용**이 핵심.
6. **구조 규칙은 추후 주입** — baseline 단계에서는 데이터로 구조(상호작용 등)를 *발견*하는 능력이 바람직.
7. **반복(iteration) 계산 시간 제약** — 알고리즘 자체의 wall-clock도 중요. 매 배치 제안 시
   **모델 적합 + 후보 탐색/획득함수 최적화**가 과도하게 오래 걸리면 안 됨(실험 주기·운영성 저해).
   특히 MAIN에서는 **하루 320개 배치를 매일** 생성하므로 모델 갱신·후보 생성이 빨라야 함.
   → N>60·데이터 누적 시 비용이 폭발하는 방식(GP-BO의 O(n³), GP-HMC 등)은 부담 →
   **데이터에 선형적으로 확장되는 모델(트리/선형/진화모델)** 을 선호.

- [ ] **최적 X 조합 탐색 파이프라인** 구축 (데이터 0건 → 반복 실험으로 Y 최대화)
- [ ] 도메인지식 없이 동작하지만, 나중에 제약/사전지식을 주입할 수 있는 **확장 가능한 구조**
- [ ] **"왜 이 방법이 baseline으로 적절한가"** 를 설득하는 산출물

### 비목표 (이번 단계에서 하지 않는 것)
- 구체적 도메인 규칙 하드코딩 (추후 단계)
- 완벽한 해석 모델 — 단, 상호작용 단서를 주는 정도의 해석성은 보너스로 추구
- 실시간/온라인 서빙

## 5. 접근 방향 (권고 baseline)

> 핵심 인식: 이것은 **"고차원(N>60) binary 조합에 대한 비싼 black-box 최적화"** 문제이며,
> **실험 예산의 체급(regime)에 따라 최적 방법이 달라진다.**
> 본 프로젝트는 **MAIN = 하루 320개(누적 O(10^4~5))**, **SUB = 하루 40개(누적 O(10^3))** 를 가정한다.

### 5.0 체급별 주력 방법
| Regime | 누적 예산 | 지배 요인 | **주력 방법** |
|--------|-----------|-----------|----------------|
| 🟢 **MAIN** (320/일) | O(10^4~5) | 구조 활용 + 알고리즘 효율 | **모델기반 연관학습 진화알고리즘(DSMGA-II/LT-GOMEA) + 지도학습 대리모델(SAEA)** |
| 🔵 **SUB** (40/일) | O(10^3) | 표본 효율 | **베이지안 최적화(BOCS형 희소회귀 / SMAC3 / TPE) + screening DoE** |

> 두 체급의 **공통 인프라**(인코딩/제약 슬롯, 데이터 스키마, 시뮬레이터, best-so-far 모니터링)는 하나로 두고,
> **옵티마이저만 교체**하는 구조로 설계한다(전환·비교 용이).

### 5.1 단계별 파이프라인

**Phase 0 — 인코딩 & 제약 표현 (도메인지식 슬롯, 공통)**
- binary → 모델 입력 표현으로 매핑. 배타/그룹은 **범주형 재파라미터화**, 조건부는 조건부 공간으로.
- 구조 규칙은 가능해지는 대로 제약/합성변수/사전지식으로 주입할 **슬롯만 비워둔다**.

**Phase 1 — 초기 데이터 확보 (cold start)**
- SUB/초기: **screening DoE**(Plackett–Burman / 분수계획 / D-optimal)로 주효과+저차 상호작용 신호 확보.
- MAIN: 첫 며칠은 DoE + 광역 무작위로 **수천 표본**을 빠르게 모아 지도학습 대리모델의 토대를 만든다.

**Phase 2 — 주력 최적화 루프 (체급별)**

- 🟢 **MAIN — 모델기반 진화 + 대리모델 보조 (SAEA)**
  - **연관학습 EA(DSMGA-II / LT-GOMEA)** 로 변수 묶음(building block) 단위 탐색
    → binary 구조·큰 상호작용에 정확히 부합하고, **학습된 linkage가 구조 추정치**가 됨(도메인지식화의 다리).
  - **지도학습 대리모델(GBM/RF)** 으로 EA가 생성한 다수 후보를 **사전선별(triage)** → 상위 320개만 실측.
  - 매일: 후보 생성 → 대리모델 선별 → 320개 실측 → 데이터 누적 → 모델·linkage 갱신.

- 🔵 **SUB — 배치 베이지안 최적화**
  - 대리모델: (해석형) **BOCS형 희소 pairwise 베이지안 회귀** / (강건형) **SMAC3(RF)·Optuna(TPE)**.
  - 획득함수: EI / **배치 Thompson sampling**(q=40에 적합). q-EI는 q>4에서 비현실적.
  - 매일: 40점 제안 → 측정 → 모델 갱신 → 반복.

**Phase 3 — 도메인지식 주입 (발전 단계, 공통)**
- 구조 규칙을 제약/사전분포(prior)/커스텀 feature/커널/EA 연관모델 초기화로 주입.
- effect heredity 등으로 상호작용/연관 탐색을 가지치기.

### 5.2 후보 도구 스택 (네트워크/OSS 사용 가능 전제)
| 용도 | 가벼운 baseline | 본격 |
|------|----------------|------|
| MAIN 옵티마이저 | GA(`DEAP`) + 휴리스틱 linkage | **DSMGA-II / LT-GOMEA(GOMEA)** 공개 구현, hBOA |
| MAIN 대리모델 | `scikit-learn` RF | **XGBoost/LightGBM** + SHAP 해석 |
| SUB BO | Optuna (TPE, ask/tell 배치) | SMAC3(RF), BoTorch/Ax, BOCS(`BOCSpy`) |
| DoE | pyDOE2 | 커스텀 D-optimal |

> **추천 출발점(MAIN)**: DoE/광역 시드 → **연관학습 EA + GBM 대리모델 선별**.
> 이유: (1) O(10^4) 예산에서 진화탐색이 본령, (2) binary 구조(building block)를 *학습*해 활용·해석, (3) 하루 320 배치와 자연스럽게 부합, (4) 트리/진화모델은 데이터에 선형 확장 → 반복시간 통제.

> ⚠️ **반복 시간 가드레일**: 매일 배치(MAIN 320, SUB 40) 생성의 wall-clock을 점검. 목표(잠정): **배치당 수 분 내**.
> GP-HMC(SAASBO)·대형 GP-BO 등 n³/HMC 비용 방법은 N>60·누적 시 부담 → 주력 제외, 비교용으로만.

### 5.3 설득 포인트
- **SUB**: 데이터 0건+점당 비용 큼 → 대량 라벨 지도학습/그리드 불가 → 능동적 BO가 본질적으로 맞음.
- **MAIN**: 예산이 10~30배 → **지도학습 대리모델과 진화탐색이 가능해짐**. 그러나 2^60은 여전히 못 훑으므로 "무작정 진화"가 아니라 **구조(linkage)를 학습·활용하는 모델기반 EA**가 정답.
- 두 체급 공통: **구조를 발견·활용**하는 모델이 추후 도메인지식 주입과 양방향으로 맞물림.
